## 一、认同的其他 Agent 观点

### 1.1 认同 DeepSeek：block bootstrap 的 block_days 应基于 empirical ACF

**理由**：IC 序列是截面 rank 相关系数的时间序列，其自相关结构取决于：
- 因子值的自相关（因子是慢变的，IC 自相关高）
- 收益率的自相关（通常较弱）
- 调仓频率（月频 IC 相邻值高度相关）

标签 horizon（20天）只决定了收益率的自相关，而 IC 的自相关远大于此。实证文献（Harvey & Liu, 2020）建议 block length 应覆盖 IC 的自相关衰减至近零的 lag。用 empirical ACF 确定 block 是正确做法。

### 1.2 认同 DeepSeek：1日标签 40 天 purge 过度保守

**理由**：1日标签的 forward return 仅 1 天，信息泄漏窗口极短。40天 purge 意味着每个测试样本丢弃 40 天的训练数据。对于日频交易（约250天/年），40天 = 丢弃 16% 的训练数据，成本过高。这确实混淆了两个不同概念：label overlap（标签重叠）和 information leakage（信息泄漏）。

### 1.3 认同 Main：concept shift yellow 误报率需要调整

**理由**：34.4% 的误报率意味着约 1/3 的 yellow 信号是噪声。这会触发大量不必要的重训，浪费计算资源并可能导致过拟合。

### 1.4 认同 Coder：实验台账应升级为 SQLite/DuckDB

**理由**：70+ 字段、数百实验的 TSV 文件，查询（"找出所有 IC>0.03 且 Sharpe>1.5 的实验"）需要手写解析。SQLite/DuckDB 提供结构化查询、索引、聚合，且 DuckDB 对分析型查询有列式存储优势。

### 1.5 认同 Main：2026 估值缺口在 holdout 窗口内是 P0 问题

**理由**：如果 2026 Q1 财报因审计延迟大面积缺失，holdout 期间的估值因子大面积 forward-fill，等于用 2025 Q3 的数据评估 2026 Q1 的策略。这不是"容忍度"问题，是数据完整性问题。holdout 的价值在于提供无偏的最终评估——如果数据大面积缺失，holdout 评估本身就失效了。

## 二、反对的其他 Agent 观点

### 2.1 反对 DeepSeek：1日标签 purge 建议 10 天——仍偏保守

**DeepSeek 立场**：1日标签 purge 应为 10 天。
**我的反驳**：

1日标签的理论最小 purge 是 1 天（标签 horizon）。但考虑到：
- 特征计算窗口：多数动量/波动率因子用 20-60 天回看窗口，但这些是 **t-1 及之前** 的数据，不包含 t 日的 return
- 市场微观结构：A股 T+1 制度下，t 日信息在 t+1 日开盘才 fully impounded
- 日历效应：周一/周五效应等不构成信息泄漏

我建议 **5 天 purge**（而非 10 天）。理由：
- 覆盖短期反转效应（1-5 天）
- 覆盖 T+1 结算延迟
- 不过度浪费训练数据（5天 vs 40天，训练数据损失从 16% 降至 2%）
- 与 de Prado (2018) 原文的 "purge = label horizon" 建议一致（加 4 天安全边际）

10天仍然是凭直觉选的数字，缺乏理论支撑。

### 2.2 反对 Main：walk-forward 实际约 180 步 vs 文档要求 24 步——这不是矛盾

**Main 立场**：walk-forward 总步数未量化，实际约180步 vs 文档要求24步，是 P0。
**我的反驳**：

Main 混淆了两个概念：
- **OOT 步数**（Out-of-Time steps）：用于最终评估，文档要求 ≥24 步
- **Walk-forward 总步数**（包括训练窗口滑动）：这是实现细节

Walk-forward 的总步数 = (总样本期 - 初始训练期) / 步长。如果步长 = 21 交易日，10 年数据约 180 步。这 180 步中，每步都产生一个 OOS 预测，但 **OOT 评估只用最后 24 个不重叠的 OOS 窗口**。

文档要求的 "24步 OOT" 指的是用于统计推断的独立 OOS 样本数，不是 walk-forward 的滑动步数。这两者不矛盾。真正的问题是：24 步 OOT 的独立样本量因训练窗口重叠（95% overlap）而小于 24——这是 DeepSeek 正确指出的问题。

**Main 的 P0 定级有误**，应降为 P1（需要澄清文档措辞，而非修复逻辑错误）。

### 2.3 反对 DeepSeek：因子方向全部预注册过于教条

**DeepSeek 立场**：所有因子必须预注册方向。
**我的反驳**：

在量化研究的探索阶段，强制所有因子预注册方向会导致：
1. **幸存者偏差的逆向**：研究者会只测试符合先验方向的因子，遗漏方向反转的 regime-dependent 因子
2. **学术先验不等于市场先验**：A股的因子方向与美股不同（如 momentum 在A股长期弱于 reversal）
3. **阻碍创新**：新因子（如另类数据因子）没有文献先验

正确的做法是 **分层管理**：
- **Tier 1（高置信）**：有 3+ 篇独立文献支撑的因子（如 value、size、momentum）→ 预注册方向，单侧检验
- **Tier 2（中置信）**：有 1-2 篇文献或 strong economic rationale → 预注册方向但保留方向翻转检查
- **Tier 3（探索性）**：新因子/另类数据 → 双侧检验，必须在独立 OOS 验证后才能纳入

这是 Main 的建议，我同意 Main 的立场。

### 2.4 反对 Review Agent（R1）：估值因子 PIT 问题被高估

**R1 我自己的原立场**：供应商日频数据可能已处理公告日时点，PIT 问题可能被高估。
**重新审视**：

在 R1 中我自己提出了这个观点，但经过进一步思考，我需要修正。A股数据供应商（如 Wind、聚源）的日频财务数据确实做了 PIT 处理，但：
1. **不是所有供应商都做 PIT**：部分免费数据源（如 Tushare）可能不做
2. **PIT 处理的质量参差不齐**：有些供应商只做了"公告日快照"但没有处理修正公告
3. **文档未指定数据供应商**：这是真正的问题——不同供应商的 PIT 处理差异巨大

因此，PIT 问题不应被降级。正确的做法是：**在文档中明确指定数据供应商，并验证其 PIT 处理逻辑**。

### 2.5 反对 Coder：Phase A0 25-45 工作日偏高

**Coder 立场**：Phase A0 纯工程实现 25-45 工作日。
**我的反驳**：

Coder 的估计假设从零开始实现所有组件。但 Phase A0 的核心是 **数据管道 + 基础设施**，不包括因子研发和策略逻辑：
- 数据采集与清洗：5-8 天（Tushare/Wind API 调用、缺失值处理、复权计算）
- Walk-forward 引擎：5-8 天（日历生成、窗口滑动、并行调度）
- 基础因子库：3-5 天（只实现文档明确列出的因子，不扩展）
- 实验台账：2-3 天（SQLite schema + 基础 CRUD）
- 监控与日志：2-3 天

合计 **17-27 工作日**，中位数约 22 天。Coder 的 45 天上限过于悲观，除非团队对 A股数据生态完全不熟悉。

## 三、其他 Agent 遗漏的重要问题

### 3.1 [P1] 停牌/除权因子处理未明确

A股停牌期间因子值如何处理？如果 forward-fill 停牌期间的因子值，复牌后因子值突变会导致：
- 停牌期间的因子排名不反映真实信息
- 复牌当日可能出现异常大的因子变动
- 如果用停牌期间的因子值生成信号，实际无法执行

文档需要明确：停牌期间是否纳入因子计算？复牌当日信号如何处理？

### 3.2 [P1] 数据版本控制与可复现性

如果上游数据源发生修正（如财务数据修正公告），如何保证历史实验可复现？
- 需要数据快照机制（point-in-time snapshot）
- 需要实验与数据版本的绑定关系
- 建议使用数据版本号 + 实验记录中的 data_version 字段

### 3.3 [P1] OOT 与 holdout 的职责分离

文档中 OOT 和 holdout 的关系不清晰：
- **OOT**：walk-forward 过程中最后 N 步的 OOS 预测，用于模型选择和超参调优
- **Holdout**：完全不参与任何决策的最终评估集

如果 OOT 用于模型选择（选择最佳因子组合/超参），那 OOT 本身就是训练过程的一部分，不能作为无偏的最终评估。需要明确：
- OOT 是否用于模型选择？如果是，holdout 的必要性更大
- Holdout 的样本量是否足够独立评估？

### 3.4 [P2] Alpha 衰减检测机制缺失

策略上线后，如何检测 alpha 因子是否衰减？文档提到 concept shift 检测，但没有区分：
- **Alpha 衰减**：因子 IC 随时间下降（市场效率提升）
- **Concept shift**：因子与收益的关系发生变化（regime change）

两者的应对策略不同：前者需要因子替换，后者需要模型重训。建议增加 alpha 衰减的专项监控指标（如滚动 6 月 IC 均值的趋势检测）。

### 3.5 [P2] 风险模型规格不够具体

文档提到"风险模型"但未明确：
- 是 Barra 类因子风险模型？还是统计风险模型（PCA）？
- 风险因子的数量和来源？
- 协方差矩阵的估计方法（sample vs. shrinkage）？

风险模型的选择直接影响组合优化的约束条件和最终持仓的行业/风格暴露。这不是 Phase A0 的问题，但在文档中应有明确规格。

## 四、关键分歧的独立立场

### 4.1 1日标签 purge

| 专家 | 建议 |
|------|------|
| 原文 | 40天 |
| DeepSeek | 10天 |
| Main | 20天 |
| **Review Agent** | **5天** |

**理由**：
- 1日标签的理论 purge = 1天（label horizon）
- 加 4 天安全边际覆盖：短期反转效应 + T+1 结算 + 微观结构噪声
- 5天损失约 2% 训练数据，40天损失 16%——这是 8 倍的数据浪费
- 与 de Prado (2018) 的建议一致（purge ≈ label horizon + small buffer）
- 10天和20天都是"折中"而非"推理"，缺乏明确的理论依据

### 4.2 Block bootstrap block_days

**立场：同意 DeepSeek，使用 empirical ACF**

补充建议：
1. 计算 IC 序列的 ACF，找到 ACF 衰减至 1/e 的 lag 作为 block length
2. 设置 block length 下限 = 21（一个交易月），上限 = 63（一个季度）
3. 对 block length 做敏感性分析：block = {21, 42, 63}，检查结论稳健性
4. 如果 empirical ACF 显示 IC 无显著自相关（lag-1 ACF < 0.1），则 block = 21 是合理的默认值

### 4.3 因子方向预注册

**立场：同意 Main 的分层方案，但补充执行细节**

分层方案需要明确的 **分类标准**：
- **Tier 1（预注册方向）**：在 3+ 篇独立论文中被验证为同方向的因子
  - 示例：EP（earnings-to-price）→ 正向（高 EP 预测高收益）
  - 检验：单侧 t-test，显著性门槛 p < 0.025（单侧等价于双侧 p < 0.05）
- **Tier 2（方向假设 + 翻转检查）**：有 1-2 篇文献支撑
  - 检验：先测预注册方向，如果不显著，测反方向（需额外扣减 multiple testing 惩罚）
- **Tier 3（探索性）**：无文献先验
  - 检验：双侧检验，显著性门槛 p < 0.01（更严格）
  - 必须在独立 OOS 验证后才能纳入

### 4.4 Concept shift 误报率

**立场：同意需要调整**

34.4% 的误报率太高。建议调整方案：
- **方案 A**：提高 yellow 门槛为 5/6 步为负（误报率 = C(6,5)×0.5^6 + C(6,6)×0.5^6 = 10.9%）
- **方案 B**：增加评估步数至 12 步，yellow = 8/12（误报率 = 19.4%）
- **方案 C**：保持 4/6 但要求连续 3 步为负（而非任意 4 步），大幅降低误报率

推荐方案 A，简单且误报率可控。

### 4.5 Newey-West 带宽

**立场：同意 DeepSeek，补充实施建议**

IC 序列的自相关结构可能复杂（AR + MA 成分），固定 lag 的 Newey-West 可能不足。建议：
1. 使用 Andrews (1991) data-dependent bandwidth 作为默认
2. 对比固定 lag（如 lag=6, 12）和 Andrews 自动带宽的结果
3. 如果两者差异 > 20%，报告差异并解释原因
4. 在稳健性检验中报告不同带宽选择下的 t-stat 分布

### 4.6 实验台账格式

**立场：同意 SQLite/DuckDB，建议 DuckDB 优先**

理由：
- DuckDB 嵌入式部署，无需 server
- 列式存储对分析查询（聚合、过滤）性能优于 SQLite
- 原生 Python/R 集成，无需 ORM
- 支持 Parquet 导出，方便与其他数据管道集成

Schema 设计建议：
```sql
CREATE TABLE experiments (
    exp_id TEXT PRIMARY KEY,
    created_at TIMESTAMP,
    factor_set TEXT,          -- 因子组合标识
    model_type TEXT,          -- LGB/XGB/NN
    label_horizon INT,        -- 标签天数
    half_life_months FLOAT,   -- 半衰期
    purge_days INT,           -- purge 窗口
    oot_sharpe FLOAT,         -- OOT Sharpe
    oot_ic_mean FLOAT,        -- OOT IC 均值
    oot_ic_std FLOAT,         -- OOT IC 标准差
    max_drawdown FLOAT,       -- 最大回撤
    turnover_monthly FLOAT,   -- 月均换手率
    status TEXT,              -- running/completed/failed
    notes TEXT
);
```
