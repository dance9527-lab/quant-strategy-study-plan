## 分歧点1：MDE功效分析 — P0还是P1？

### 回应Main的挑战

Main 的核心论点："OOT 是 smoke test，不是精确功效工具。24 步只是最低验收门槛，keep 决策基于完整 OOT（~180 步）。"

**我部分接受这个论点，但指出一个关键遗漏。**

Main 说得对：keep 决策基于完整 OOT，而非 24 步。24 步的定位是 smoke test——排除明显的假阳性。在这个定位下，低功效（对弱 alpha）是可以接受的，因为 smoke test 的目标不是"检测所有 alpha"，而是"排除没有 alpha 的情况"。

**但是，Main 回避了一个文档层面的实际问题：文档中存在多处基于 24 步的决策规则，且未标注这些规则的功效边界。**

具体来说：
1. **§9.1 的 65% 方向一致性门槛**：文档说"exploratory tracking 要求 65% 的 OOT 步方向一致"。如果这是基于全部 OOT 步（~180步），功效充足。但如果团队实际操作中在 24 步后就计算方向一致性来决定是否继续，功效严重不足。
2. **§6.1.5 的 concept shift 5/6 门槛**：这个规则在 6 步窗口上判断，本质上与 24 步 smoke test 面临相同的功效问题。

**我的让步**：如果文档明确将 24 步定位为"仅排除 Sharpe > 1.5 的假阳性"，并标注所有基于 24 步的规则为"启发式，非统计检验"，那么 MDE 可以从 P0 降级为 P1。

### 回应Review的挑战

Review 的核心论点："24 步不是 keep 决策依据，MDE 分析只影响结果解读，不阻塞 S1 启动。"

**我同意这个逻辑框架。** P0/P1 的区分标准是"是否阻塞 S1 启动"。如果 MDE 分析只影响对 smoke test 结果的解读方式，那确实是 P1。

**但 Review 漏掉了一个阻塞性场景**：如果团队不知道 24 步 smoke test 的功效边界，可能会在 24 步 smoke test "不通过"时错误地放弃一个实际上有 alpha 的策略（假阴性决策）。这不是"结果解读"问题，是"决策规则设计"问题。

**具体场景**：
- 一个 Sharpe = 0.6 的 alpha（IC ≈ 0.0104），24 步 OOT 的功效约 20%
- 24 步 smoke test 结果：IC 均值 = 0.008（t-stat = 0.65，不显著）
- 如果团队不知道功效只有 20%，可能在 24 步后判断"无 alpha"并停止追踪
- 实际上，如果继续追踪到 180 步，t-stat = 0.008 / (0.06/√180) = 1.79，p = 0.037，显著

**这个场景是否阻塞 S1 启动？** 取决于文档如何定义 24 步 smoke test "不通过"后的处理流程：
- 如果"不通过"→ 等待更多步数 → 不阻塞（P1）
- 如果"不通过"→ 放弃追踪 → 阻塞（P0）

**Review 和 Main 都假设前者，但文档未明确。** 这就是我认为需要补充的内容。

### 我的最终立场

**从 P0 调整为 P1+（P1 的高优先级，接近 P0）。**

理由：
1. 我接受 Main 和 Review 的核心论点：keep 决策基于完整 OOT，24 步是 smoke test
2. 但我坚持：文档必须明确 24 步 smoke test 的功效边界和"不通过"后的处理流程
3. 如果文档补充了以下内容，MDE 降级为标准 P1：
   - 功效分析表（不同 Sharpe 对应的 24 步/180 步功效）
   - 24 步 smoke test 的定位："仅排除 Sharpe > 1.5 的假阳性"
   - "不通过"后的处理规则："smoke test 不通过应等待更多步数，不得基于 24 步做出放弃决策"

### 说服论据

**致 Main 和 Review**：你们的论点在统计学上是正确的——smoke test 不需要高功效。但你们忽略了一个实际风险：**如果团队不知道 smoke test 的功效边界，可能会过度依赖 smoke test 结果做出放弃决策。** 功效分析表的作用不是"精确估计 alpha"，而是"告诉团队 smoke test 能做什么、不能做什么"。这是一个文档完整性问题，成本极低（一次计算），但能防止错误的决策规则设计。你们愿意接受"在 §5.5 增加功效分析表 + 明确 24 步定位"作为 P1+ 方案吗？

## 分歧点4：Sleeve FDR计入 — P0还是P1？

### 回应Main的挑战

Main 的核心论点："sleeve 不独立训练模型，共享同一个 alpha 信号，只是持仓分层不同，对 FDR 影响可控。"

**这个论点在统计学上是错误的，让我给出具体计算。**

Main 的逻辑是："共享模型 → 不是独立实验 → 不应独立计入 FDR"。但 FDR 校正的输入不是"独立训练的模型数量"，而是"独立的假设检验数量"。即使 5 个 sleeve 共享同一个模型，如果每个 sleeve 独立做出 keep/not-keep 决策，那就是 5 个独立的假设检验。

**类比**：一个研究者用同一个数据集检验 5 个不同的子假设（例如，同一回归模型中 5 个不同系数是否显著）。即使数据来源相同、模型相同，这 5 个检验仍然需要多重比较校正。BH 校正关心的是"做了多少次检验"，不是"用了多少个模型"。

### BH 阈值收紧的具体计算

**场景设定**：
- 基础实验族（无 sleeve）：因子(15) × 标签(3) × 模型(7) × 半衰期(3) × 正交化(2) × 执行(2) = **3,780 次尝试**
- 含 5 个 sleeve：3,780 × 5 = **18,900 次尝试**（保守估计，假设并非所有维度都与 sleeve 交叉，取 ~12,600）

**BH 校正公式**：第 k 小的 p 值需满足 p_(k) ≤ (k / m) × α，其中 m = 实验总数，α = 目标 FDR（通常 0.10）。

**对最显著假设（k=1）的 BH 阈值**：

| 场景 | 实验总数 m | BH 第1名阈值 | BH 第10名阈值 | BH 第100名阈值 |
|---|---|---|---|---|
| 不含 sleeve | 3,780 | 2.65×10⁻⁵ | 2.65×10⁻⁴ | 2.65×10⁻³ |
| 含 5 个 sleeve | 12,600 | 7.94×10⁻⁶ | 7.94×10⁻⁵ | 7.94×10⁻⁴ |
| **收紧倍数** | **3.3×** | **3.3×** | **3.3×** | **3.3×** |

（注：我 R2 报告中说"收紧 5 倍"是基于 3,780→18,900 的完整交叉。实际中如果并非所有维度都与 sleeve 交叉，收紧倍数约为 3-5 倍。取保守值 3.3 倍。）

**关键问题**：收紧 3.3 倍意味着什么？

假设在不含 sleeve 的情况下，有 20 个因子的 p 值在 0.003-0.01 之间（刚好通过 BH 校正）。如果阈值收紧 3.3 倍（从 0.01 降到 0.003），这 20 个因子中有大部分会被淘汰。

**这不是"可控"的影响——这是对 alpha 保留率的实质性改变。**

### 回应Main的"sleeve共享test_family_id"方案

Coder 提出的方案："sleeve 共享 test_family_id，不独立成族。" Main 支持这个方案。

**这个方案能部分缓解问题，但不能完全解决。**

分析：
- 如果 sleeve 共享 test_family_id，那么 BH 校正在整个 family 内执行，attempt_count 包含所有 sleeve 的尝试
- 这意味着 BH 阈值确实会收紧（因为 attempt_count 增加了）
- 但如果团队的意图是"sleeve 不应影响 FDR"，共享 test_family_id 不能实现这个意图——它只是让 FDR 校正在更大的实验族上执行

**真正的分歧是**：sleeve 的尝试是否应该计入 attempt_count？
- 如果"是"：FDR 阈值收紧 3-5 倍，这是正确的统计实践
- 如果"否"：FDR 阈值不变，但整体假阳性率被低估

**Main 和 Coder 的方案（共享 test_family_id）实际上接受了"是"的立场**——因为共享 family_id 意味着 attempt_count 包含 sleeve 尝试，BH 阈值自然收紧。他们可能没有意识到这一点。

### 我的最终立场

**维持 P0，但接受 Coder 的方案作为实施方式。**

理由：
1. 所有 sleeve 的尝试必须计入 attempt_count（这是统计学要求）
2. Coder 的"共享 test_family_id"方案正确实现了这一点
3. 但文档必须明确披露："本次实验族包含 N 个 sleeve，BH 校正后的 p 值阈值为 X"
4. 如果团队认为 3-5 倍收紧过于严格，应减少 sleeve 数量（如只保留 baseline + 1-2 个核心 sleeve），而非跳过 FDR 校正

### 说服论据

**致 Main**：你说"sleeve 不独立训练模型，对 FDR 影响可控"——但你自己支持的方案（Coder 的共享 test_family_id）实际上会让 attempt_count 增加 3-5 倍，BH 阈值收紧 3-5 倍。这与"影响可控"矛盾。要么接受 FDR 收紧是正确代价，要么减少 sleeve 数量。你不能同时说"影响可控"和"共享 family_id"。

**致 Coder**：你的共享 test_family_id 方案在实现上是正确的，但你说"这是实现细节，不阻塞框架设计"——我不同意。FDR 计入规则直接影响 alpha 保留率，是框架设计的核心参数，不是实现细节。文档必须在 §3.1.1 中明确规则，否则不同团队成员会做出不同的 FDR 假设。

## 分歧点5：Concept shift独立性 — P0还是P1？

### 回应Main的挑战

Main 的核心论点："补充说明性段落即可。正自相关降低误报率，5/6 门槛更保守，为什么还是 P0？"

**Main 的逻辑是**：误报率降低了（从 10.9% 降到 2-5%）→ 5/6 门槛更保守 → 不需要修改 → P1。

**这个逻辑忽略了功效问题。**

我已经在 R2 中修正了方向错误：正自相关**降低**误报率，而非升高。Main 正确采纳了这个修正。但 Main 的结论"补充说明即可"是基于一个不完整的分析——只看了误报率，没看功效。

**功效不足的具体影响**：

1. **有效样本量减少**：在 AR(1) 下，有效样本量 n_eff = n × (1-ρ)/(1+ρ)。对于 n=6, ρ=0.3：n_eff = 6 × 0.7/1.3 ≈ 3.2 步。

2. **检测真实 concept shift 的功效下降**：
   - 假设 IC 从 +0.03 降至 -0.01（策略真正失效），Δ = -0.04，σ_IC = 0.06
   - 独立假设下：6 步中≥5步 IC<0 的概率 ≈ 45%（功效尚可）
   - ρ=0.3 下：6 步中≥5步 IC<0 的概率 ≈ **15-20%**（功效严重不足）

3. **这意味着什么？** 如果策略真正失效（concept shift 发生），CSRP 告警系统只有 15-20% 的概率检测到。**80-85% 的概率会漏报。**

4. **为什么漏报比误报更危险？**
   - 误报（假阳性）→ 触发告警 → 团队检查 → 发现无问题 → 取消告警。成本：一次检查的时间。
   - 漏报（假阴性）→ 不触发告警 → 策略在 regime change 中继续运行 → 持续亏损。成本：策略失效期间的全部亏损。

**"补充说明性段落"不解决功效问题。** 需要的是：
1. 在文档中明确标注功效风险
2. 建立功效监控机制（OOT 报告中同时报告 IC 自相关系数）
3. 预定义功效不足时的应对方案（缩短窗口或降低门槛）

### 回应Main的"修正方向后问题变小了"论点

Main 暗示：既然误报率从"升高"修正为"降低"，问题的严重性降低了。

**恰恰相反。** 让我解释：

- R1 的问题（误报率升高）→ 解决方案：收紧门槛（如 6/6）。简单直接。
- R2 修正后的问题（功效降低）→ 解决方案：需要功效分析 + 替代方案设计。更复杂。

功效问题比误报率问题更难解决，因为：
1. 误报率可以通过收紧门槛简单解决（代价是功效进一步降低）
2. 功效问题需要重新设计检测机制（缩短窗口、降低门槛、增加步数），每个方案都有副作用

**所以修正方向后，问题不是变小了，而是变复杂了。**

### 我的最终立场

**维持 P0。**

理由：
1. 功效不足是系统性设计缺陷，不是"说明性段落"能解决的
2. 漏报（80-85% 的概率漏掉真实 concept shift）比误报更危险
3. 需要在文档中建立功效监控机制和预定义应对方案
4. 这些修改影响 CSRP 告警系统的设计，必须在 S1 启动前完成

### 说服论据

**致 Main**：你说"补充说明即可"——请问，如果一个告警系统有 80% 的概率漏掉真实的风险事件，你会只在文档里"补充说明"就上线吗？功效不足不是文档润色问题，是告警系统的核心能力缺陷。你需要回答一个问题：**如果 concept shift 发生了，但 CSRP 告警系统检测不到（80% 概率），这个后果由谁承担？**

**致 Coder**：你说"自相关检测和 block bootstrap 是现有框架的自然扩展，不阻塞设计"——我同意实现成本低，但你忽略了一个关键点：**如果不在文档中预定义功效不足时的应对方案，OOT 阶段发现功效不足时团队会临时决策，这比预先设计更危险。** 预定义方案（缩短窗口到 4 步、降低门槛到 4/6）需要在 S1 启动前写入文档，否则就是在没有预案的情况下上线告警系统。
