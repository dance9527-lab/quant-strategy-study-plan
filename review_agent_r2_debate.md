### 分歧点1：MDE功效分析 — P0还是P1？

**我的立场**：**P1**

**对其他agent的反驳/支持**：

- **支持 Main R1** 的结论（P1），但**不完全同意其理由**。Main 说"OOT 是 smoke test，不是精确功效工具"——这个理由过于轻描淡写。即使 OOT 是 smoke test，如果连 smoke test 都无法检测到经济上有意义的 alpha，那 smoke test 的结果解读方式需要调整。Main 的理由应该修正为："OOT 24 步是最低验收门槛，完整 OOT 约 180 步才是正式验证；功效分析应在完整 OOT 框架下进行，而非针对 smoke test 门槛。"

- **驳斥 DeepSeek R1** 的 P0 立场。DeepSeek 的功效计算（IC 均值 0.015 时 t-stat=1.22）是正确的，但结论不正确。原因有三：
  1. **24 步不是 keep 决策的依据**。文档已明确"24 步只是最低验收门槛，不是总步数"、"完整 S1-M OOT 步数按 calendar 计算并披露，当前粗略预期约 180 步"。keep 决策基于完整 OOT（约 180 步），此时 IC 均值 0.015 对应 t-stat = 0.015 / (0.06 / √180) ≈ 3.35，功效充足。
  2. **MDE 分析的价值是解释性的，不是阻塞性的**。知道"24 步 smoke test 对 IC < 0.025 的 alpha 检测功效不足 80%"是有用信息，但它不阻塞 S1 启动——它只影响对 smoke test 结果的解读方式（例如，smoke test 不通过时需要等待更多步数，而非立即放弃）。
  3. **P0 的定义是"阻塞 S1 启动"**。MDE 分析不阻塞 S1 启动，只影响结果解读。按此标准应为 P1。

- **部分支持 Coder R1** 的"P0 但成本低"观点。成本低是事实，但"成本低"不等于"应该 P0"。P0/P1 的区分标准是"是否阻塞 S1 启动"，不是"实现成本"。如果用实现成本决定优先级，所有"1 天能做完"的任务都会变成 P0，P0 就失去了意义。

**我的定义/方案**：

MDE 分析应在 Phase A0.2 阶段完成（不阻塞 S1 启动），作为 smoke test 结果解读的参考框架。具体要求：
1. 给定 24 步 OOT、IC 标准差 σ_IC（从历史数据估计）、显著性水平 α=0.05（单侧）、功效 1-β=0.80，计算最小可检测 IC 均值 MDE_24
2. 同样计算 180 步 OOT 的 MDE_180
3. 在实验报告中注明："当前 smoke test（24 步）的 MDE 为 X，完整 OOT（约 180 步）的 MDE 为 Y。若 smoke test 不通过但 Y < 预期 IC，建议等待更多步数。"
4. MDE 分析不需要在每个实验中重复执行——一次计算后作为常数写入 validation_params.json

**如果被说服，说明原因**：我被 DeepSeek 关于功效计算的具体数字说服了（24 步对弱 alpha 确实功效不足），但我对结论（P0）不认同，因为 24 步不是 keep 决策的依据。

### 分歧点2：CSRP命中窗口 — P0还是P1？如何定义？

**我的立场**：**P0**（R1 立场不变，但被 DeepSeek 说服增加组合收益命中标准）

**对其他agent的反驳/支持**：

- **驳斥 Main R1** 的 P1 立场。Main 说"主口径=标签持有期，敏感性10/40日"——这个定义过于笼统。问题在于：
  1. CSRP 信号是 concept shift 告警，不是 alpha 信号。它的"命中"含义应该是"信号发出后，因子/策略确实恶化了"，而不是"信号发出后，标签收益为负"。
  2. 如果用标签持有期（20 日）作为命中窗口，一个 concept shift 信号可能在 20 日内被市场噪声淹没，导致假阴性率极高。
  3. 命中窗口应基于 OOT step（月频）而非交易日，因为 CSRP 信号本身就是按 OOT step 触发的（"最近 6 步中至少 5 步 IC < 0"）。

- **支持 DeepSeek R1** 的 P0 立场和组合收益命中标准。DeepSeek 正确指出，仅用 IC 符号作为命中标准忽略了效应大小。一个 OOT step 的 IC 为 -0.001（统计上不显著）和 IC 为 -0.05（经济上显著）应区别对待。组合收益（成本后超额收益）是更好的命中标准，因为它同时捕获了方向和大小。

- **支持 Review R1** 自己的 P0 理由：命中窗口是误报率框架的前提参数，没有命中窗口就无法计算 FPR。这不是"细节"，是框架完整性问题。

**我的定义/方案**：

| 参数 | 定义 | 理由 |
|------|------|------|
| **窗口长度** | 信号触发后 3 个成熟 OOT step（约 3 个月） | CSRP 信号基于月频 OOT IC，命中判定应在相同频率上进行；3 步覆盖约 3 个月，足以区分噪声和真实 concept shift |
| **命中标准（主口径）** | 3 步中至少 2 步成熟 IC 与信号方向一致（IC < 0）且组合成本后超额收益 < 0 | 双重标准：IC 方向确认 + 经济意义确认 |
| **命中标准（敏感性）** | 3 步中至少 2 步 IC < 0（仅 IC，不加组合收益约束）| 更宽松的敏感性口径 |
| **排列检验** | 10,000 次 block permutation（保留信号的时序聚类结构）| DeepSeek 正确指出简单 shuffle 破坏了 CSRP 信号的自相关结构 |
| **最小信号数** | `n_signals >= 5` 才可报告 FPR 点估计；`n_signals >= 20` 才可作为 tighten-only 依据 | 小样本下排列检验功效极低 |
| **预注册** | 命中窗口和命中标准必须在实验登记中预注册，不得根据结果择优 | 防止数据窥探 |

**如果被说服，说明原因**：我被 DeepSeek 说服将命中标准从"仅 IC 方向"扩展为"IC 方向 + 组合收益"。DeepSeek 的论据——"仅 IC 符号忽略效应大小"——是正确的。但我保持 P0 立场不变，因为命中窗口是 FPR 框架的前提参数。

### 分歧点3：裁剪与overlay执行顺序

**我的立场**：**P0**，执行顺序为：**裁剪 → 归一化到 1.0 → 乘 capital_multiplier**

**对其他agent的反驳/支持**：

- **支持 Main R2** 的顺序（裁剪→归一化→overlay），但**不完全同意其表述**。Main 说"先裁剪→归一化→overlay"，但未明确归一化的目标值。如果归一化到 1.0（满仓），再乘 capital_multiplier（如牛市 0.9），最终权重 = 0.9，现金 = 0.1。这是正确的。

- **驳斥 Review R1**（即我自己在 R1 中的建议）的表述歧义。R1 中我说"分层裁剪在 capital overlay 之前执行，裁剪后权重乘以 capital_multiplier"——这个表述跳过了归一化步骤。实际执行顺序应该是：

  1. **第一步：分层裁剪**（总换手 → 行业 → 个股），每层等比例缩减
  2. **第二步：归一化到 1.0**（裁剪后权重总和可能 < 1.0，因为部分权重被裁剪为 0；归一化确保满仓基准）
  3. **第三步：乘 capital_multiplier**（将归一化后的满仓权重乘以市场状态对应的 multiplier，如牛市 0.9、熊市 0.2）

  **为什么这个顺序正确**：
  - 如果先乘 multiplier 再裁剪：假设 multiplier = 0.2（熊市），总权重 = 0.2，然后裁剪总换手上限 10%。此时总换手上限 = 0.2 × 10% = 0.02，过于严格。
  - 如果裁剪后不归一化直接乘 multiplier：裁剪后权重总和可能是 0.85（15% 被裁剪），乘以 0.9 后 = 0.765，但实际目标是"满仓的 90%"= 0.9。归一化后再乘才正确。

- **支持 DeepSeek R1** 的"P0"立场，四位一致。

**我的定义/方案**：

```
步骤1: 理想调仓权重 w_raw（从模型输出）
步骤2: 分层裁剪
  2a: 总换手裁剪（等比例缩减至总换手上限）
  2b: 行业裁剪（等比例缩减至行业上限）
  2c: 个股裁剪（等比例缩减至个股上限）
  每层裁剪后重新检查后续层是否仍超限
步骤3: 归一化 w_clipped = w_clipped / sum(w_clipped)  [确保 sum = 1.0]
步骤4: 乘资本乘数 w_final = w_clipped × capital_multiplier
步骤5: 现金 = 1 - sum(w_final)
```

**如果被说服，说明原因**：我被自己的 R1 分析说服，补充了归一化步骤。R1 中的表述遗漏了这一步，但逻辑方向正确。

### 分歧点4：Sleeve FDR计入规则 — P0还是P1？

**我的立场**：**P0**（R1→R2 立场变化：从 P1 修正为 P0）

**对其他agent的反驳/支持**：

- **支持 DeepSeek R1** 的 P0 立场和核心论据。DeepSeek 指出 5 个 sleeve 使实验总数从 ~2,520 增至 ~5,880-12,600，这个计算是正确的。让我验证：

  基础实验族（无 sleeve）：因子(~15) × 标签(3: 1d/5d/20d) × 模型(7) × 半衰期(3: 等权/12月/18月) × 训练窗口(1) × 正交化分支(2) × 执行规则(2) = 15 × 3 × 7 × 3 × 1 × 2 × 2 = 3,780

  加入 5 个 sleeve：如果每个 sleeve 独立计入 attempt_count，则 3,780 × 5 = 18,900。但实际中 sleeve 主要影响组合构建阶段，不改变因子/标签/模型/半衰期，所以更合理的计算是：因子(~15) × 标签(3) × 模型(7) × 半衰期(3) × sleeve(5) × 正交化(2) × 执行(2) = 18,900。

  DeepSeek 给出的 5,880-12,600 范围比我计算的保守，可能是因为他假设并非所有维度都与 sleeve 交叉。无论如何，**sleeve 的加入使实验族显著增大**，BH 校正的 p 值阈值会更严格。

- **驳斥 Main R2** 的 P1 理由。Main 说"sleeve 不独立训练模型，对 FDR 影响可控"——这个论据有两个问题：
  1. **FDR 不关心是否独立训练模型**。FDR 关心的是"尝试了多少种不同的假设"。每个 sleeve 都是一个独立的假设（"P80-P95 分段是否有 alpha？"），即使它们共享同一个模型。BH 校正的输入是 p 值列表和假设总数，不是模型数量。
  2. **"影响可控"未量化**。Main 没有给出 BH 校正后的 p 值阈值变化。让我计算：假设无 sleeve 时实验族大小 N=3,780，BH 在 α=0.10 下的第 k 个 p 值阈值 = k/N × 0.10。加入 5 个 sleeve 后 N'=18,900，阈值 = k/N' × 0.10。对于 k=1（最显著的假设），阈值从 0.10/3,780 ≈ 2.65e-5 降至 0.10/18,900 ≈ 5.29e-6。**阈值收紧了 5 倍**，这不是"可控"的影响。

- **修正 Review R1** 的 P1 立场。R1 中我说"区分诊断性 sleeve 和候选策略 sleeve"，这个区分仍然有价值，但不足以将 FDR 影响降为 P1。即使诊断性 sleeve 不计入 keep 决策，它们仍然消耗 attempt_count，影响 BH 校正。

**我的定义/方案**：

| Sleeve 类型 | 是否计入 FDR | 理由 |
|-------------|-------------|------|
| `baseline_top_score` (Top-N 基线) | ✅ 计入 | 这是主策略，必须计入 |
| `upper_middle_score_p80_p95` | ✅ 计入 | 候选策略 sleeve，独立假设 |
| `small_mid_mv_p20_p80_top_score` | ✅ 计入 | 候选策略 sleeve |
| `balanced_mv_p20_p80_score_p60_p95` | ✅ 计入 | 候选策略 sleeve |
| `micro_mv_p0_p20_diagnostic_only` | ⚠️ 部分计入 | 诊断性 sleeve，不计入 keep 决策，但计入 attempt_count |

**关键规则**：
1. 所有 sleeve 的尝试必须记录在 `attempt_count` 中
2. 只有 `baseline_top_score` 的 keep 决策使用包含所有 sleeve 的 FDR 校正
3. 候选策略 sleeve 的 keep 决策独立于 baseline，但也必须通过 FDR 校正
4. 在实验报告中必须披露："本次实验族包含 N 个 sleeve，BH 校正后的 p 值阈值为 X"

**如果被说服，说明原因**：我被 DeepSeek 的定量论据说服了。BH 校正阈值收紧 5 倍是显著影响，不能用"可控"一笔带过。R1 中我的"诊断性 vs 候选策略"区分仍然有价值，但不能改变 FDR 会计的基本规则。

### 分歧点5：Concept shift独立性假设 — P0还是P1？

**我的立场**：**P0**

**对其他agent的反驳/支持**：

- **完全采纳 DeepSeek R2 的数学修正**。DeepSeek 正确指出：正自相关（ρ > 0）会**降低**零假设下的误报率，而非升高。这是因为正自相关使连续 6 步 IC 同号的概率更高（无论正负），所以"6 步中 5 步为负"在正自相关下反而更不容易出现——除非真正的均值已经偏移。

  数学验证：假设 IC_t = ρ × IC_{t-1} + ε_t，其中 ε_t ~ N(0, 1-ρ²)。
  - ρ = 0 时：P(6 步中 ≥5 步 IC < 0) = C(6,5)×0.5^6 + C(6,6)×0.5^6 = 7/64 ≈ 10.9%
  - ρ = 0.3 时：连续 6 步同号的概率更高，但"连续 6 步为负"需要均值确实为负。如果零假设下 IC 均值为 0，正自相关使序列更"粘"在当前状态，一旦 IC 跳到正区间，更难在 6 步内全部回到负区间。DeepSeek 估算 ρ=0.3 时误报率降至 ~2.5%，方向正确（具体数值取决于模型细节）。

- **驳斥 Main R2** 的 P1 理由。Main 说"补充说明性段落即可"——这个建议过于轻描淡写。原因：
  1. **DeepSeek 的修正改变了问题的性质**。原问题（R1）是"正自相关可能升高误报率，需要 block bootstrap 校正"。修正后的问题变为"正自相关降低误报率，但降低功效"。这是**不同性质的问题**，需要不同的解决方案。
  2. **功效不足才是真正的风险**。如果正自相关使 6 步 IC 序列更"粘"，那么即使真正的 concept shift 发生了，6 步内可能只有 3-4 步为负（因为之前的正 IC 拖累了序列），无法触发 yellow 告警。这意味着 **CSRP 的灵敏度被高估了**。
  3. "补充说明性段落"不解决功效问题。需要的是：(a) 用 block bootstrap 估计实际功效，(b) 如果功效不足，考虑缩短判定窗口或降低触发门槛。

- **支持 Coder R2** 的 P0 立场和 block bootstrap 方案。Coder 正确指出应该用 block bootstrap 估计实际误报率和功效。但 block bootstrap 的 block 长度需要校准——建议 block = 21 日（与月频 OOT step 对齐）。

- **支持 Review R1（即我自己）** 的 P0 立场，但理由需要更新。R1 的理由是"正自相关升高误报率"，这被 DeepSeek 修正为"正自相关降低误报率但降低功效"。P0 结论不变，但理由变了。

**我的定义/方案**：

1. **文档必须注明**：5/6 yellow 门槛的 10.9% 理论误报率基于 IC 步间独立假设。实际 walk-forward 中 IC 存在正自相关（因训练窗口重叠），这会**降低**零假设误报率但**降低** concept shift 检测功效。

2. **必须进行 block bootstrap 功效分析**：
   - block = 21 日（与月频 OOT step 对齐）
   - 重抽样次数 ≥ 5,000
   - 估计在不同 IC 自相关水平（ρ = 0, 0.1, 0.2, 0.3）下，5/6 门槛检测真实 concept shift（IC 均值从 0.03 降至 -0.01）的功效
   - 如果功效 < 0.80，必须在实验报告中注明并讨论替代方案

3. **替代方案（如果功效不足）**：
   - 缩短判定窗口：从"最近 6 步"改为"最近 4 步"（4 步中 3 步为负触发 yellow）
   - 降低触发门槛：从 5/6 改为 4/6
   - 以上替代方案必须作为敏感性报告，不得替换主口径

4. **P0 理由**：功效分析直接影响 concept shift 告警系统的设计。如果功效不足，告警系统会产生大量假阴性（真实 concept shift 不被检测），这比假阳性更危险——因为假阳性只触发报告，而假阴性意味着策略在 regime change 中继续运行而不被告警。这是一个**系统性设计缺陷**，不是"说明性段落"能解决的。

**如果被说服，说明原因**：我被 DeepSeek 的数学修正说服，修正了 R1 中关于"正自相关升高误报率"的错误认知。但我保持 P0 立场不变，因为修正后的问题（功效不足）比原问题（误报率偏高）更严重——误报率偏高只导致过度告警（保守偏误），功效不足导致漏报（激进偏误）。
