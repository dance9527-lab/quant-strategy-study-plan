## A股量化策略研究计划独立评审
> 审计时间：2026-04-30 04:35 CST
> 审计人：Main Agent (AutoClaw)

## 整体评分：6.5/10

**一句话总结**：数据底座工程扎实、验证框架设计严谨，但计划过于理想化，缺乏执行约束和时间线，在实际落地时大概率会经历大幅延期或需要多次裁剪。

---

## 核心发现（Top 5）

### 1. 数据底座得分高，但"最后一公里"缺口被低估
warehouse 已完成6轮审计+P1前置，15类目录全部PASS，工程完成度约85%。但剩余缺口（停牌公告、ST、指数成分、成本假设）对策略研究的影响被系统性低估：
- 停牌只有工程推断+2023年AkShare，历史推断准确率未量化
- ST只有深市历史，沪/北缺失意味着约40%股票的风险警示状态不完整
- 指数成分只有当前快照，任何需要历史指数增强的研究都无法进行

**建议**：在S1启动前，先量化每个缺口对因子IC的影响（如：有/无ST过滤的因子IC差异）。

### 2. S1通过门槛可能造成"永远无法通过"困境
S1要求同时满足：
- RankIC t-stat > 2
- bootstrap p < 0.05
- 12 OOT steps
- holdout不失效
- 成本后超额为正
- 年化换手 < 300%

这些条件单独看都合理，但组合起来非常严格。如果核心因子IC只有0.02-0.03（这在A股日频是常见水平），在5年训练/21日调仓的walk-forward下，t-stat可能很难超过2。

**建议**：增加一个"弱alpha但稳健"的通过路径，如IC方向一致+分年度正率>70%+成本后不为负，标记为"candidate"而非直接discard。

### 3. 风险开关100/60/30/0缺乏理论和实证基础
总纲中提出"牛市100%、震荡60%、熊市30%、极端0%"的仓位开关，但：
- 这些数字从何而来？没有任何回测或文献支持
- "牛市/震荡/熊市"的定义本身就有大量自由度
- 在walk-forward中验证意味着需要提前定义市场状态，这本身就是一个建模决策

**建议**：先做市场状态分类的稳定性研究（不同定义下的状态持续期、转换概率），再讨论仓位比例。

### 4. 容量测试时机不当
S1要求同时覆盖1000万/5000万/1亿的容量测试，但此时连因子是否有效都不知道。容量测试应该在S1证明存在可交易alpha后作为S2的深化研究。

**建议**：S1只做最简单的成交额过滤（如日均成交额>1000万），S2再做精细容量分析。

### 5. 模型路线过于保守，可能错过快速筛选机会
计划中的模型顺序是：单因子→等权→ICIR→Ridge→LightGBM→Ranker。这个顺序虽然稳健，但在A股日频研究中，LightGBM通常能比线性模型快10倍发现非线性关系。

**建议**：在单因子IC筛选的同时并行跑LightGBM，用SHAP分析快速发现有效特征组合，而不是串行等待。

## 逐项审计意见

### A. 数据底座：8/10

**优点**：
- PIT合规性设计到位：return_adjusted_pit、available_at、tradability_daily三层保障
- 行业分类已支持12个标准的PIT区间
- 泄漏检查覆盖15类目录

**问题**：
1. valuation_daily与收益表key不一致（2026-01-05至2026-02-05缺口）未说明原因
2. 行业分类"尝试5834只，成功5510只，空返回/错误324只"——这324只的分布（退市vs在市）未说明
3. 复权因子"最后日期晚于最后一根日K bar的有255只"——这些证券的收益计算是否正确处理？

### B. 验证框架：7/10

**优点**：
- walk-forward + purge + embargo的设计是教科书级别的
- split label audit是很好的创新
- 过拟合审计（DSR/PBO/holdout）框架完整

**问题**：
1. purge >= max(horizon, 20)对于20日标签意味着每月至少丢弃1个月数据，在5年训练窗口中损失约17%样本
2. embargo 5天是否足够？A股的T+1意味着信息传播可能需要更长
3. "12个OOT steps"每个21天=252天≈1年，只覆盖1个完整年度周期
4. 没有讨论如何处理极端事件期间（如2015股灾、2020疫情）的特殊处理

### C. 工程可行性：5/10

**问题**：
1. **无时间线**：Phase A/B/C/D各需要多久？S1到S7的依赖关系是什么？无法评估
2. **算力约束**：Intel Arc B390 16.5GB能跑LightGBM walk-forward（可行），但TFT/N-HiTS/PatchTST基本不可能（至少需要24GB+显存）
3. **并行度**：6 worker并行处理warehouse构建是合理的，但walk-forward的并行化未讨论
4. **代码管理**：只提到Git，但实验结果的版本管理、模型checkpoint的存储策略未讨论

### D. 风险控制：6/10

**优点**：
- 涨跌停买卖失败的统计指标设计好
- 连续锁死和解锁后反转的监控是A股特色
- 成本分层（L1/L2/L3）思路清晰

**问题**：
1. 风险开关的具体触发条件未定义（"市场宽度崩塌"如何量化？）
2. 熔断的冷却期和恢复条件未定义
3. 没有讨论黑天鹅事件的处理（如突然停牌、退市、政策突变）
4. 没有讨论因子拥挤风险（当多个策略使用相同因子时）

### E. 缺失项

1. **无回撤控制机制**：除了风险开关外，没有讨论策略层面的止损/减仓规则
2. **无因子正交化方案**：多个因子高度相关时如何处理？
3. **无交易执行细节**：T+1执行的具体规则（开盘价vs VWAP vs 收盘价）未明确
4. **无监控告警系统**：策略运行后的监控和告警机制缺失
5. **无退出/放弃标准**：什么情况下应该放弃一个策略方向？

## 改进建议（按优先级排序）

1. **P0：增加时间线和里程碑**：每个Phase/S步骤应有估计工期和依赖关系
2. **P0：放宽S1"弱alpha"路径**：增加candidate状态，不要全部discard
3. **P0：量化数据缺口影响**：在S1启动前，先做有/无ST过滤、有/无停牌推断的因子IC对比
4. **P1：风险开关先做基础研究**：市场状态分类的稳定性分析应早于仓位比例讨论
5. **P1：并行化LightGBM筛选**：不要串行等待单因子IC完成
6. **P1：定义退出标准**：什么条件下放弃一个方向
7. **P2：容量测试延后**：S1只做简单成交额过滤，S2再精细分析
8. **P2：深度模型可行性评估**：在Phase D前先评估硬件是否支持

## 结论

这个计划的"骨架"是好的——数据底座扎实、验证框架严谨、禁止事项明确。但"肌肉"还不够——缺乏执行约束、时间线和灵活性。建议：
1. 先聚焦S1（日频多因子基线），用最简单的模型验证warehouse下是否存在可交易alpha
2. 放宽通过门槛，增加candidate状态
3. 量化数据缺口影响，而不是只披露
4. 风险开关从简单规则开始，不要一开始就追求100/60/30/0的精确仓位
