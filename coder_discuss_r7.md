# Coder R7 讨论 — 工程架构视角

## 角色：代码架构和工程实现专家

---

## 1. 回应各方审计结论

### 1.1 回应 Review（9.0/10）

Review 给出高分，认为双主线设计完整。从架构设计角度看，双主线确实是一个清晰的抽象——S1-M 和 S1-D 共享相同的 pipeline 骨架（数据→特征→训练→选股→回测），只是参数不同。**抽象层面我同意 Review 的判断。**

但 Review 提到的 `daily_orders_audit` 和 `daily_turnover_capacity_report` 尚未建立，这正是我上次评分偏低的核心原因。这两个组件不是可选的——它们是 S1-D 能否上线的**硬性前置条件**。

### 1.2 回应 DeepSeek（4.9/10）

DeepSeek 的观点最激进，但**工程上我部分同意**：

- **交易成本问题**：DeepSeek 说年化约 50%，Main 算出 75.6%。无论哪个数字，结论一致——**日调仓在成本上不可行**。这一点我完全同意。
- **过拟合风险**：每日重训模型，训练窗口滚动快，样本量相对不足，过拟合风险确实高于月频。工程上需要额外的正则化和验证机制。
- **但"双主线增加复杂度"这一点我不同意**：如果 S1-M 和 S1-D 共享 pipeline 骨架，额外复杂度主要在参数配置层，而非代码架构层。好的抽象不应该因为"有两个用例"就变得复杂。

**我的立场：DeepSeek 对业务逻辑的判断（成本、过拟合）是正确的，但对工程架构的判断（复杂度）过于悲观。**

### 1.3 回应 Main（7.0/10）

Main 的分析最务实：

- **21 倍换手率**：这个数字直接说明了问题。月调仓换手率如果在 30-50%，日调仓就是 630-1050%，加上 A 股的印花税+佣金，年化成本 75.6% 完全合理。
- **1日/5日标签信噪比低**：这是模型层面的问题，工程上可以通过更严格的特征筛选和集成学习来缓解，但无法根本解决。
- **S1-D 的价值在于验证短期因子**：这个定位我认同。S1-D 不应该是一个生产系统，而应该是一个**实验平台**。

## 2. 保留 S1-D 需要多少额外工程量？

从工程角度，S1-D 相对于 S1-M 的**增量工程**如下：

| 组件 | 工作量（人天） | 说明 |
|------|--------------|------|
| daily_orders_audit | 3-5 | 日频订单审计，检查滑点、成交偏差、异常换手 |
| daily_turnover_capacity_report | 2-3 | 日频换手率容量报告，计算实际可执行规模 |
| 日频数据 pipeline | 2-3 | 日频特征计算、标签生成（1d/5d） |
| 日频回测引擎适配 | 1-2 | 复用月频引擎，但需处理 T+1、涨跌停等日频细节 |
| 日频风控模块 | 2-3 | 日频止损、最大持仓天数、单日最大换手限制 |
| 日频监控 dashboard | 1-2 | 实时/日终监控面板 |
| **合计** | **11-18 人天** | |

**关键发现**：如果 S1-M 和 S1-D 共享 pipeline 骨架，增量工程量可控（约 2-3 周）。但如果骨架设计不好，耦合过紧，增量可能翻倍。

## 3. 我是否仍然坚持"先 S1-M 后 S1-D"？

**是的，但立场有调整。**

上一轮我说"先 S1-M 后 S1-D"，隐含假设是 S1-D 最终会成为生产系统。经过本轮讨论，我修正为：

> **S1-M 是生产主线，S1-D 降级为探索性实验平台。**

理由：
1. **成本不可行**：75.6% 年化成本意味着 S1-D 永远无法作为独立生产系统运行
2. **信噪比不足**：1d/5d 标签噪声大，模型可靠性存疑
3. **但实验价值存在**：短期因子是否有效、日频信号能否增强月频选股——这些问题值得回答

---

## 4. 具体建议：工程上如何平衡 S1-M 和 S1-D？

### 4.1 架构设计：共享骨架 + 参数化差异

```
┌─────────────────────────────────────┐
│         Pipeline Orchestrator        │
│  ┌─────────────┐ ┌─────────────┐    │
│  │   S1-M Config│ │ S1-D Config │    │
│  │  freq: month │ │  freq: day  │    │
│  │  label: 20d  │ │  label: 1d  │    │
│  │  retrain: 1M │ │  retrain: 1D│    │
│  └──────┬──────┘ └──────┬──────┘    │
│         │               │            │
│  ┌──────▼───────────────▼──────┐     │
│  │   Shared Pipeline Engine     │     │
│  │  数据→特征→训练→选股→回测   │     │
│  └─────────────────────────────┘     │
│                                       │
│  ┌─────────────────────────────┐     │
│  │   S1-D Only: 实验模块       │     │
│  │  · daily_orders_audit       │     │
│  │  · turnover_capacity_report │     │
│  │  · 短期因子有效性验证       │     │
│  └─────────────────────────────┘     │
└─────────────────────────────────────┘
```

### 4.2 实施顺序

| 阶段 | 内容 | 周期 |
|------|------|------|
| Phase 1 | S1-M 完整 pipeline（生产级） | 4-6 周 |
| Phase 2 | Pipeline 骨架参数化改造 | 1-2 周 |
| Phase 3 | S1-D 实验平台搭建 | 2-3 周 |
| Phase 4 | S1-D 实验运行 + 因子验证 | 持续 |

### 4.3 S1-D 实验平台的核心约束

S1-D 不是生产系统，但需要以下工程约束：

1. **成本上限**：单日换手率 cap 在 20%，超出则自动降频
2. **模型冻结窗口**：不是每日重训，而是每周重训 + 每日微调（仅更新最后一层）
3. **实验报告自动生成**：每次实验自动输出 Sharpe、换手率、成本、因子 IC
4. **与 S1-M 的因子共享机制**：S1-D 验证有效的短期因子可以注入 S1-M 的特征集

### 4.4 风险控制

| 风险 | 缓解措施 |
|------|---------|
| S1-D 过拟合 | OOT 验证 + Walk-forward + 每周重训而非每日 |
| 工程复杂度膨胀 | 严格参数化，S1-D 不得引入 S1-M 不存在的模块 |
| 成本失控 | 硬性 cap + 自动降频机制 |
| 与 S1-M 资源竞争 | S1-D 优先级低于 S1-M，资源不足时暂停 |
